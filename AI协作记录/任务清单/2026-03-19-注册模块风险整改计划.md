# 注册模块风险整改计划（GPT-5.4 执行版）

## Summary
本轮目标不是“补一个能用的注册页”，而是把注册模块从原型态提升到可交付态，重点一次性解决 3 类问题：  
1. 匿名用户可抢注并绑定任意成员身份  
2. 前端页面在未登录时先返回敏感项目数据再跳登录  
3. 注册/认证改造后，全量测试仍未收口

本轮采用已确认方案：**当前长期方案先做邮箱验证，不做短信验证**。  
为保证邮箱验证真的能防止冒名注册，本轮同步引入 **成员邮箱作为权威绑定字段**：只有邮箱与成员档案一致且完成验证，才能完成自助注册。

## Key Changes
### 认证与注册策略
- 将注册流程改为 **“成员邮箱匹配 + 邮箱验证 + 激活登录”**，不再允许匿名用户仅凭 `member_id` 直接绑定成员。
- 为成员增加权威邮箱字段；只有已配置邮箱的成员才允许自助注册。
- 注册成功后**不自动登录**，而是返回“待验证”状态；只有邮箱验证完成后才允许登录并创建 session。
- 登录接口拒绝未验证邮箱的账号，并返回明确但不泄露内部细节的错误语义。
- 保留超级账号体系，但超级账号不走成员自助注册流程；默认由管理入口创建。

### 数据与接口
- `members` 增加 `email` 字段，唯一、可为空；本轮作为成员实名绑定依据。
- `accounts` 增加：
  - `email`
  - `email_verified_at`
  - `registration_status`（默认 `pending_verification`，验证后为 `active`）
- 新增邮箱验证令牌表，建议独立为 `email_verification_tokens`：
  - `id`
  - `account_id`
  - `email`
  - `token_hash`
  - `expires_at`
  - `consumed_at`
  - `created_at`
- `/auth/register` 入参改为至少包含：`login_id`、`password`、`member_id`、`email`
- 新增 `/auth/verify-email`，消费验证 token，激活账号
- 新增 resend 能力：`/auth/resend-verification`
- `/auth/login` 仅允许 `registration_status=active` 且 `email_verified_at` 非空的账号登录
- `/auth/me` 响应中增加账号验证状态，供前端显示

### 页面与权限
- `/`、`/overview`、`/project/{project_id}` 统一改成服务端先做会话与可见性校验，再决定是否返回页面。
- 工作台和概览页只返回当前 session 可见范围内的数据；项目详情页必须校验项目可见权限。
- 登录页与注册页补齐“待验证邮箱”提示、重新发送验证邮件入口、验证成功跳转逻辑。
- 若当前环境没有真实邮件服务，先接入**开发态邮件发送抽象**：生产实现走 SMTP/邮件 API，测试环境走内存捕获/假发送器。

## Test Plan
- 注册成功但邮箱未验证：账号创建成功、无 session、状态为 `pending_verification`
- 注册邮箱与成员档案邮箱不一致：拒绝注册
- 成员未配置邮箱：拒绝自助注册
- 重复注册、重复邮箱、重复 login_id：拒绝
- 验证 token 过期、重复消费、伪造 token：拒绝
- 邮箱验证成功后首次登录：成功创建 session
- 未验证账号登录：明确失败
- 匿名访问 `/`、`/overview`、`/project/{id}`：不返回敏感页面内容
- 已登录但无权访问项目详情：403/拒绝访问
- 依赖保存与删除相关前端测试：不再断言 `current_member_id` 契约
- 全量回归：`python -m pytest -q` 通过
