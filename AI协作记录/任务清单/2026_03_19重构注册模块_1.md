账户社交化与项目邀请重构实施计划书
Summary
本次改造的目标是把系统明确拆成两套互不混淆的体系：

超级账户体系：只用于你本地模拟测试，固定为 god / 888888，可切换全部虚拟身份，不参与普通用户社交关系
普通账户体系：注册即创建唯一业务身份，普通账户之间可按账户ID搜索、关注、互关成好友，并通过“项目邀请确认”加入项目
这份计划以“后续加功能不伤筋动骨”为核心约束，设计上坚持：

产品统一账户ID，但数据库内部不强行绑定 Account.id == Member.id
社交关系围绕 Account 建模
项目协作关系围绕 Project + Invitation 建模
超级测试体系与普通社交体系物理隔离
不复用老的 members 匿名查询逻辑来承载社交功能
Architecture Principles
统一边界
认证/注册边界：只负责账户创建、邮箱验证、登录会话、普通账户唯一成员身份初始化
成员资料边界：只负责普通账户绑定成员的个人资料，不承载社交关系
社交关系边界：只负责搜索、关注、互关、好友分类，全部以 Account.id 为主键
项目邀请边界：只负责“谁邀请谁加入哪个项目、当前状态是什么”，不直接和关注表混写逻辑
超级账户边界：只负责模拟切换和测试，不进入普通用户搜索、关注、好友、邀请流程
后续扩展的稳定点
不让“关注关系”直接等同“项目关系”；以后要加黑名单、备注、分组、好友申请都能继续往社交模块扩展
不让“项目成员表”直接承载邀请状态；以后要加提醒、过期、撤回、历史记录都能继续往邀请模块扩展
不把“超级测试身份”塞进普通账户体系；以后要删掉测试身份或换成脚本注入，不影响普通社交
普通用户界面只认 Account.id；以后内部怎么调整 Member 表都不影响对外交互

Ordered Tasks
Task 0：先做结构止损与边界清理
目标：先把“普通账户社交”和“超级测试体系”在设计上分开，避免后面越改越缠
主要改动

明确约束：普通搜索/关注/好友/邀请全部只面向普通账户
明确超级账户与虚拟身份不进入普通社交
明确普通用户产品层统一展示 Account.id
不再让注册页依赖 /members/ 匿名拉取成员
输出
重构设计基线
新模块命名与职责边界敲定
完成标准
后续任务都不再需要碰“超级账户是否进社交体系”这种决策
Task 1：重构普通用户注册流
目标：把普通注册改成“注册即建唯一身份”
主要改动

POST /auth/register 改入参为 email / password / username / gender
注册时自动：
创建普通 Account
创建对应 Member
建立一对一绑定
发邮箱验证
移除普通公开注册中的 member_id 绑定逻辑
超级账户公开注册继续禁止
边界要求
超级账户创建方式不走公开注册
注册成功仍是待邮箱验证，不自动登录
完成标准
普通用户无需先有成员，即可完成注册
普通用户注册完成后有自己的唯一成员身份
Task 2：简化成员资料模型
目标：把普通成员资料从“内部管理数据”调整成“用户资料”
主要改动

Member.tel 改为非必填
新增 gender
新增 public_email、public_tel
用户名允许重名
创建成员时最少只需名字，性别默认 private
边界要求
不强行重写项目业务里所有 Member 相关逻辑
只调整必要校验与展示层
完成标准
成员资料支持轻量化创建
不再以手机号作为普通用户注册前提
Task 3：建立独立社交关系模块
目标：把关注/好友能力做成独立子系统
主要改动

新增 account_follows 模型
新增社交服务层（建议独立 social_service）
实现：
关注
取消关注
关系分类聚合
账户ID搜索普通用户
搜索结果只返回公开资料
边界要求
不复用 members.py 作为社交查询接口
不让项目模块直接算好友关系，统一通过社交服务获取
完成标准
甲可按账户ID搜到乙
甲可关注乙
乙回关后自动成为好友
任一方取消关注后自动降级
Task 4：新增 /social 独立页面
目标：给普通用户一个统一的社交操作入口
主要改动

新增 /social 页面与模板
页面包含 5 个区块：

我关注谁
谁关注我

Task 5：建立项目邀请子系统
目标：把“拉人入项目”改成标准邀请确认流程
主要改动

新增 project_invites 模型
新增邀请服务层（建议独立 project_invite_service）

发起人（PM）
被邀请普通账户
状态

边界要求
项目成员表只记录最终成员，不记录邀请中间态

PM 发邀请不等于对方已入组

Task 6：给项目页补邀请入口
目标：让 PM 在真实项目流里能拉人
主要改动

两种邀请方式都支持：

输入任意普通账户ID邀请

被邀请人未接受前，不出现在项目正式成员列表
边界要求
非 PM 不显示邀请入口
快捷好友邀请只是 UI 便利，本质仍走统一邀请接口
完成标准
甲不是乙好友时，也能用账户ID邀请乙
不管是否好友，乙都必须接受后才入项

Task 9：补回归测试与迁移测试
目标：把这次改造做成可持续迭代基础

Task 10：最后做一轮接口与页面收口
目标：统一用户心智，避免未来继续散乱
主要改动

后续如果加黑名单、备注、好友分组、项目角色，只需要在社交/邀请模块扩展
Suggested Implementation Boundaries
为保证后续扩展稳定，建议新增而不是硬塞进旧模块：

app/api/social.py：搜索、关注、关系列表
app/services/social_service.py：关注关系与好友分类
app/api/project_invites.py：项目邀请相关接口
app/services/project_invite_service.py：邀请创建、接受、拒绝
app/templates/social.html：独立社交页
现有 app/api/auth.py：只收注册字段和绑定初始化逻辑
现有 app/api/projects.py：只保留项目业务，调用邀请服务，不自己处理邀请状态
现有 app/api/members.py：只保留成员资料管理，不承担社交检索

Assumptions and Defaults
产品层统一使用 Account.id 作为普通用户公开ID
数据库内部保留 Member，但不强制 Account.id == Member.id
普通用户只有一个唯一业务身份
超级账户仅作模拟，不参与普通社交关系
关注不需要确认；项目邀请必须确认
用户名允许重名，唯一识别靠账户ID
性别固定三值：男 / 女 / 保密
邮箱和手机默认私密，可单独公开
取消关注默认立即打破好友关系
