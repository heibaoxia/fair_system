# 项目现状总结

> [!info]
> 本总结基于 2026-03-18 对仓库代码、`docs/TODO.md`、当前工作区变更和测试结果的阅读整理。
>
> 后续指令记录：[[指令发布记录]]

## 一句话判断

项目已经明显过了“搭基础骨架”的阶段，当前主线是：`项目创建 -> 自定义评分 -> 评分进度追踪 -> 汇总综合分 -> 公平分配 -> 交付审核 -> 项目结算`。现在更像功能整合和产品打磨期。

## 我看到的核心业务闭环

1. 工作台创建项目，创建时必须配置 `scoring_dimensions`，也可以顺带创建模块和依赖关系。
2. PM 为项目设置评分期，项目成员只在评分窗口内提交评分。
3. 打分页只负责按项目既定维度打分，不允许临时改权重。
4. 评分完成后，系统按动态维度计算模块综合分，并支持批量公平分配。
5. 模块负责人开发后上传成果，PM 审核通过后模块才能真正完成，并解锁依赖模块。
6. 项目全部完成后，按综合分比例做项目结算并累计到成员 `total_earnings`。

## 当前已经比较成型的能力

### 1. 动态评分体系已经落地

- 每个项目独立拥有 `ScoringDimension`
- 创建项目时校验至少 1 个维度、权重和约等于 `1.0`、维度名不能重复
- `ModuleAssessment` 不再直接存旧式固定字段，而是通过 `DimensionScore` 记录每个维度的明细分数
- 项目详情页、打分页、综合分汇总都在读同一套项目评分配置

### 2. 评分流程控制已经成型

- PM 可设置评分开始时间与持续时长
- 成员只有在评分期内才能提交
- 评分进度接口会区分 `进行中 / 已完成 / 已失效`
- `effective_completion` 已作为后续汇总和分配的关键闸门

### 3. 分配与依赖联动已经打通

- 公平分配前会先检查评分是否已有效完成
- 批量分配会参考近 30 天负载和模块综合分
- 模块依赖支持单条/批量保存，并有环检测
- 未被依赖解锁的模块不能上传成果

### 4. 页面层功能已经不少

- `index.html`：工作台、项目卡片、项目创建表单
- `project_detail.html`：项目画板、成员管理、模块管理、依赖编辑、评分期、分配预览
- `scoring_page.html`：成员按项目动态维度逐模块打分
- `todo.html`：待评分项目聚合、交换请求、在做模块、钱包汇总
- `timeline.html`：项目进度看板
- `members.html`：成员管理

## 当前开发阶段的明显信号

- `docs/TODO.md` 显示你这轮核心目标“项目自定义评分维度 + 动态渲染 + 打分页移除旧权重逻辑”已经完成
- 当前待办更偏向产品增强：登录验证、项目创建后的评分维度编辑方案、更多端到端测试、总窗口继续完善
- Git 工作区仍在持续开发中：当前有多处已修改文件，集中在 `projects/scoring/frontend/templates/tests` 这一批核心文件

补充观察：

## 我认为当前最值得注意的风险/欠账

> [!warning]
> 下面这些不是主流程阻塞，但会影响后续继续扩展。

1. `app/templates/project_detail.html`、`app/templates/index.html` 已经非常大，页面逻辑和业务逻辑耦合偏重，后续维护成本会越来越高。
2. `app/services/calculator.py` 仍在读取 `ModuleAssessment.estimated_hours`，但当前模型里这个字段已经不存在，说明这里很可能是旧版本遗留代码。
3. `app/api/frontend.py` 的时间线页面里 `actual_hours` 还是估算/演示逻辑，不是真实工时采集。
4. 数据库变更目前主要靠 `create_all` + 手写 `ALTER TABLE` 补字段，还没有正式迁移机制。
5. 权限体系仍偏原型期：很多接口依赖 `current_member_id` 传参，且保留 `member_id = 0` 的“上帝视角”。
6. 仓库根目录有一个异常条目 `nul`，这类保留名文件在 Windows 环境里需要特别小心处理。

## 我对现阶段代码质量的判断

- 主流程已经能跑，且已有一批针对关键行为的测试保护
- 评分体系相关改造已经基本贯通到模型、接口、页面、测试四层
- 目前最大的风险不是“功能完全没做完”，而是“继续叠功能时会越来越难维护”

## 下一阶段更合理的推进顺序

1. 先把登录/身份体系补上，替代 `member_id` 手动切换和“上帝视角”
2. 再整理前端大模板，把超大页面拆成更清晰的组件或脚本模块
3. 补端到端测试，重点覆盖 `创建项目 -> 打分 -> 分配 -> 上传审核 -> 结算`
4. 最后再考虑“项目创建后允许编辑评分维度”的方案
